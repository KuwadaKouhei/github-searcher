# Test Philosophy

> 本プロジェクトのテストの拠り所。「どう配分し・何を・いつ・どうモックし・何を目的にテストするか」を体系化する。
> 具体のテスト項目は REQUIREMENTS.md §3.6 / DETAIL_DESIGN.md §5、全体思想は DESIGN_PHILOSOPHY.md §5 を参照。

---

## 0. 貫く一文

**カバレッジの数値ではなく「壊れたら困る境界」に確信を置く。** テストはリファクタを妨げる足かせではなく、安心して変更するための土台にする。

---

## 1. 採用する思想（5つの軸）

テスト思想は「配分・対象・タイミング・モック方法・目的」という別々の問いへの答えの組み合わせ。各軸で次を採用する。

| 軸 | 問い | 採用 |
| --- | --- | --- |
| 配分 | どの種類をどれだけ書くか | テスティングトロフィー |
| 対象 | 何を検証するか | 振る舞いをテスト（実装詳細でない） |
| タイミング | いつ書くか | 部分的TDD（アダプタはテストファースト） |
| モック | どう隔離するか | デトロイト学派寄り（MSWで境界モック） |
| 目的 | 何のためか | カバレッジ非目的（重要な境界に厚く） |

---

## 2. 配分：テスティングトロフィー

- 配分は下から **静的解析 > ユニット > インテグレーション（最厚）> E2E（少）**。
- スローガン: 「テストは書く。多すぎず。ほとんどインテグレーション。」
- **なぜピラミッドでなくトロフィーか**: フロントエンドのバグの多くは部品同士のつなぎ目（操作→状態→表示）で起きる。ユニットを積み上げても「ユーザーの操作が通る」確証になりにくい。ユーザーの使い方に近いインテグレーションテストが、1テストあたりの確信度／コストが最良。
- **静的解析を土台に含める**: TypeScript（`strict`）とESLintで、バグを書いた瞬間に潰す。これも広義のテストの一層と捉える。

---

## 3. 対象：振る舞いをテストする

- **指針**: 「テストはソフトウェアの使われ方に似ているほど確信を与える」（Testing Library guiding principle）。
- 内部state・関数名・クラス名といった**実装詳細を検証しない**。`getByRole` / `getByLabelText` / `getByText` で、ユーザーから見た振る舞いを検証する。`data-testid` は最終手段。
- **なぜか**: 実装詳細に紐づくテストはリファクタしただけで壊れ、変更の足かせになる。振る舞いに紐づけば、内部をどう書き換えても振る舞いが同じなら通り続ける。
- 「テストが壊れやすい＝設計が悪いシグナル」と捉え、テストの壊れやすさを設計改善の手がかりにする。

---

## 4. タイミング：部分的TDD（テストファースト）

- 全面的なTDDは課さない。ただし**ドメイン型とアダプタはテストファースト**で書く（先に失敗するテスト→実装）。
- **なぜアダプタか**: アダプタは純粋関数で書きやすく、かつ「APIとUIの契約」という壊れたら最も困る境界。ここを先にテストで固定すると、契約が明文化され実装がぶれない。
- **テスト容易性を設計の制約として先に効かせる**: 表示コンポーネントは props のみ受け取る純粋な部品にし、データ取得を混ぜない。これによりフェッチは lib（アダプタ・クライアント）で、表示は props を渡してテスト、と確実に分離できる。

---

## 5. モック：デトロイト学派寄り（MSWで境界モック）

- できるだけ本物を使い、モックは**ネットワーク境界のみ**に絞る。**MSW（Mock Service Worker）** で GitHub API への通信を傍受する。
- **なぜか**: `fetch` を直接スタブすると実装に結合し、本番と乖離する。境界でモックすれば、実際のクライアントコード（`lib/github/client.ts`）をそのまま走らせられ、本番に近いまま検証できる。
- 外部システム（GitHub）という本当に必要な箇所だけをモックし、内部の部品は本物を使う。

### 5.1 背景：二つの学派

「依存先を本物のまま使うか、モックに差し替えるか」には歴史的に二つの立場がある。

| | ロンドン学派（mockist） | デトロイト学派（classicist） |
| --- | --- | --- |
| 方針 | テスト対象を**孤立**させ、依存先を全てモックに差し替える | できるだけ**本物**を使い、モックは必要最小限の境界のみ |
| 利点 | 失敗箇所の特定が速い（依存先は全部偽物） | 本番の動きに近く「本当に動く」確信が高い。実装に結合せずリファクタに強い |
| 欠点 | 本物同士を繋いだとき動く保証がない。モックが実装の内部に結合し、リファクタで壊れやすい | 失敗時の原因特定がやや粗い |

### 5.2 本プロジェクトの立場（デトロイト学派寄り）

- 検索フローの結合テストでは、**本物のコンポーネント・本物のアダプタ・本物のクライアントを繋いだまま**動かす。偽物に差し替えるのは**ネットワーク境界の一箇所だけ**（MSWがGitHub APIへのHTTP通信を傍受）。
- **なぜ境界だけか**: GitHub API は外部システムで、テストから実際に叩けない（遅い・不安定・レート制限・結果が変動）。「本物を使えない正当な理由がある境界」がここ。逆に、自分のコード（アダプタ・クライアント）をモックにする正当な理由はない — 本物が速く安定して動くなら本物の方が確信が高い。
- **「振る舞いをテストする」（§3）との一貫性**: モックの呼び出し検証（「この引数で1回呼ばれた」）は実装詳細の検証であり、内部の呼び方を変えただけで壊れる。本物を繋いで「画面に結果が出た」を検証すれば、内部をリファクタしても振る舞いが同じならテストは通り続ける。デトロイト学派寄りの選択と振る舞いベースの検証は、同じ価値観（リファクタ耐性・本番への確信）の二つの現れ。
- **「寄り」の意味**: 宗派ではなく程度の問題。必要なら部分的なスタブを使う柔軟性は残す。原則は「デフォルトは本物、モックは境界だけ、置くならネットワークという外縁に」。

---

## 6. 目的：カバレッジを目的にしない

- 網羅率の**数値目標は掲げない**。数値を満たすための価値の低いテスト量産を避ける。
- カバレッジは「テストされていない危険な箇所」を見つける指標としてのみ使う。
- **「壊れたら困る境界」に厚く置く**。本アプリでは アダプタ（契約）＞ APIクライアントのエラー写像 ＞ 検索フロー ＞ 詳細描画 の順で重要。

---

## 7. 何をテストするか / しないか

### テストする（壊れたら困る順）

1. **アダプタ（純粋関数・ユニット・テストファースト）**: APIとUIの契約。変換・`language:null`・`watchers=subscribers_count`・URLスキーム検証・数値欠損フォールバック。
2. **APIクライアントのエラー写像**: MSWで 403/404/422 を返し、型付きエラーへ変換されるか。
3. **検索フロー結合テスト（トロフィーの中心）**: submit→結果 / 0件→空 / 403→エラー / view切替。
4. **詳細描画テスト**: 7項目・ラベル・a11y、外部リンクの `href`/`rel`。

### テストしない（YAGNIをテストにも適用）

- Next のルーティング内部、shadcn/ui の挙動そのもの、具体的なCSSクラス名。
- E2E は本課題では任意。余力があれば Playwright で「検索→詳細」のハッピーパス1本のみ。多層に重ねない。

---

## 8. ツール

- **Vitest**（ランナー）/ **React Testing Library**（振る舞い検証）/ **@testing-library/jest-dom** / **MSW**（境界モック）。
- 実行環境は jsdom。CIで lint・typecheck・test を必須にする。

---

## 9. RSCのテストに関する注意（現実的事情）

- Server Component を直接レンダリングするテストはエコシステムが発展途上。
- 対策: フェッチを表示コンポーネントから分離する（§4）。フェッチは lib（アダプタ・クライアント）でテスト、表示は props を渡して Testing Library でテスト、と確実に分ける。

---

## 10. 生きたドキュメントとしてのテスト

- 具体的なテストケースは仕様書として別途書かず、テストコードの `describe` / `it` の説明文に語らせる（二重管理を避ける）。
- 例: `it('watcherには watchers_count ではなく subscribers_count を使う')`。テスト実行結果の一覧がそのまま仕様の一覧になり、実装と乖離しない。
